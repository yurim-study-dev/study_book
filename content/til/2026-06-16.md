## 오전: JOIN이란? - 3.1

### 조인이 필요한 이유

관계형 데이터베이스는 데이터 중복을 막고 무결성을 지키기 위해 테이블을 쪼개서 저장하는 **정규화** 과정을 거친다. JOIN은 이렇게 나뉜 여러 테이블을 다시 하나의 결과로 묶어서 조회하는 핵심 기법이다.

```
정규화: 하나의 정보를 여러 테이블로 분리 저장 (중복 방지, 무결성 유지)
JOIN  : 분리된 테이블을 다시 묶어서 완전한 정보로 조회
```

- 여러 번 쿼리를 날리는 대신 한 번의 JOIN으로 처리 → 네트워크 비용 절감, 생산성 향상
- 기준 테이블의 각 행에 대해 대상 테이블에서 조건(주로 `ON` 절)에 맞는 행을 찾아 연결

---

### 예시로 이해하는 JOIN 작동 원리

**post 테이블**

| id | member_id | title |
|----|-----------|-------|
| 1 | 1 | 안녕하세요 |
| 2 | 2 | 질문 있어요 |
| 3 | 1 | 두번째 게시글 |
| 4 | NULL | 설치 오류가 났어요 |

**member 테이블**

| id | name |
|----|------|
| 1 | 하루 |
| 2 | 나무 |
| 3 | 용쌤 |

`post.member_id = member.id` 기준으로 조인할 때:
- 게시글 1, 2, 3번은 회원과 매칭됨
- 게시글 4번은 작성자가 없어서(member_id가 NULL) 매칭 안 됨
- 회원 3번(용쌤)은 작성한 글이 없어서 매칭 안 됨

| 조인 방식 | 결과 |
|-----------|------|
| **INNER JOIN** | 매칭되는 1, 2, 3번 게시글만 결과에 포함. 4번 게시글과 용쌤은 제외 |
| **LEFT OUTER JOIN** | 1, 2, 3번 + 매칭 안 된 4번 게시글도 포함 (회원 정보는 NULL로 채움) |
| **RIGHT OUTER JOIN** | 1, 2, 3번 + 매칭 안 된 용쌤도 포함 (게시글 정보는 NULL로 채움) |

```
INNER JOIN       : 양쪽 다 매칭되는 행만           → 교집합
LEFT OUTER JOIN  : 왼쪽 테이블 전체 + 매칭되는 오른쪽 → 왼쪽 기준 전체
RIGHT OUTER JOIN : 오른쪽 테이블 전체 + 매칭되는 왼쪽 → 오른쪽 기준 전체
```

## 오전: INNER JOIN 실습 - 3.2

두 테이블에서 조인 조건이 **일치하는 행만** 결과에 포함한다. 한쪽이라도 매칭이 안 되면 그 행은 결과에서 제외된다.

```sql
SELECT 컬럼목록
FROM 테이블A
INNER JOIN 테이블B ON 테이블A.컬럼 = 테이블B.컬럼;

-- 별칭(Alias) 사용
SELECT 컬럼목록
FROM 테이블A AS a
INNER JOIN 테이블B AS b ON a.컬럼 = b.컬럼;
```

```sql
-- 게시글과 작성자 이름을 함께 조회 (회원 정보가 있는 게시글만 조회됨)
SELECT post.id, post.title, member.name AS writer_name, post.created_at
FROM post
INNER JOIN member ON post.member_id = member.id;

-- 별칭(Alias)으로 간결하게 표현
SELECT p.id, p.title, m.name AS writer_name, p.created_at
FROM post p
INNER JOIN member m ON p.member_id = m.id;

-- 3개 테이블 조인: 댓글 + 작성자 + 원본 게시글 제목
SELECT reply.id, reply.content, member.name AS writer_name, post.title AS post_title
FROM reply
INNER JOIN member ON reply.member_id = member.id
INNER JOIN post ON reply.post_id = post.id;
```

> JOIN을 여러 번 이어붙이면 3개 이상의 테이블도 한 번에 묶어서 조회할 수 있다. (`reply` → `member`, `reply` → `post` 각각 연결)

## 오후: OUTER JOIN - 3.3

조인 조건이 일치하지 않아도 한쪽 또는 양쪽 테이블의 데이터를 포함하는 기법.

### LEFT OUTER JOIN - 왼쪽 테이블 전체 포함

왼쪽 테이블의 모든 행을 포함하고, 매칭되는 오른쪽 데이터가 없으면 `NULL`로 채운다.

```sql
-- 전체 회원과 각 회원의 게시글 수 조회 (게시글 없는 회원도 포함)
SELECT m.id, m.name, COUNT(p.id) AS post_count
FROM member m
LEFT JOIN post p ON m.id = p.member_id
GROUP BY m.id, m.name;

-- 전체 게시글과 댓글 수 조회 (댓글 없는 게시글도 0으로 표시)
SELECT post.id, post.title, COUNT(reply.id) AS reply_count
FROM post
LEFT JOIN reply ON post.id = reply.post_id
GROUP BY post.id, post.title;

-- 작성자 없는 게시글도 포함 + IFNULL로 '익명' 표시
SELECT p.id, p.title, IFNULL(m.name, '익명') AS writer_name, p.created_at
FROM post p
LEFT JOIN member m ON p.member_id = m.id;
```

### RIGHT OUTER JOIN - 오른쪽 테이블 전체 포함

LEFT JOIN의 반대. 실무에서는 가독성을 위해 테이블 순서를 바꿔서 **LEFT JOIN으로 통일**하는 경우가 많다.

```sql
-- RIGHT JOIN
SELECT m.id AS member_id, m.name, p.title
FROM post p
RIGHT JOIN member m ON p.member_id = m.id;

-- 위와 동일한 결과: 테이블 순서를 바꾼 LEFT JOIN
SELECT m.id AS member_id, m.name, p.title
FROM member m
LEFT JOIN post p ON p.member_id = m.id;
```

### FULL OUTER JOIN - MySQL은 직접 지원 안 함

양쪽 테이블 모두 전체 포함하고 매칭 안 되면 NULL. MySQL은 `LEFT JOIN`과 `RIGHT JOIN` 결과를 `UNION`으로 합쳐서 우회 구현한다.

```sql
SELECT m.id AS member_id, m.name, p.title
FROM member m
LEFT JOIN post p ON m.id = p.member_id
UNION
SELECT m.id AS member_id, m.name, p.title
FROM member m
RIGHT JOIN post p ON m.id = p.member_id;
```

## 오후: SELF JOIN - 3.4

하나의 테이블을 자기 자신과 조인하는 기법. 동일 테이블이 두 번 등장하므로 **반드시 별칭으로 구분**해야 한다.

추천인 관계, 사원-상사 관계, 댓글-대댓글처럼 같은 테이블 안에 계층 구조가 있을 때 사용한다.

```sql
-- member 테이블에 추천인(recommender_id) 컬럼 추가, 자기 자신을 참조하는 FK 설정
ALTER TABLE member ADD COLUMN recommender_id INT;
ALTER TABLE member ADD CONSTRAINT fk_member_recommender
  FOREIGN KEY (recommender_id) REFERENCES member(id) ON DELETE SET NULL;

-- 추천인 등록 (나무의 추천인은 하루, 용쌤의 추천인은 나무)
UPDATE member SET recommender_id = 1 WHERE id = 2;
UPDATE member SET recommender_id = 2 WHERE id = 3;

-- 회원 이름과 그 추천인의 이름을 함께 조회 (같은 테이블을 m, r 두 별칭으로 구분)
SELECT m.name AS member_name, r.name AS recommender_name
FROM member m
INNER JOIN member r ON m.recommender_id = r.id;
```

## 오후: 집합 연산자 (UNION, UNION ALL) - 3.5

여러 `SELECT`문 결과를 하나로 합치는(합집합) 연산자. **컬럼 개수와 타입이 일치해야** 결합 가능하다.

### UNION - 중복 제거

```sql
-- 1번 회원과 2번 회원이 작성한 게시글 제목 합치기 (중복 제목 제거됨)
SELECT title FROM post WHERE member_id = 1
UNION
SELECT title FROM post WHERE member_id = 2;
```

### UNION ALL - 중복 포함, 더 빠름

```sql
-- 중복 검사/정렬 작업 없이 그대로 합침 → UNION보다 속도 빠름
SELECT title FROM post WHERE member_id = 1
UNION ALL
SELECT title FROM post WHERE member_id = 2;
```

| 연산자 | 중복 처리 | 성능 |
|--------|----------|------|
| `UNION` | 중복 제거 (내부 정렬 발생) | 느림 |
| `UNION ALL` | 중복 그대로 유지 | 빠름 |

## 오후: 서브쿼리(Subquery) - 4

하나의 SQL 문 안에 포함된 또 다른 `SELECT` 문. 복잡한 조건이나 중간 연산을 단계적으로 처리할 때 사용한다.

### 4.1 WHERE 절 서브쿼리

```sql
-- 단일 행 서브쿼리: 결과가 1개 → 비교 연산자(=) 사용
SELECT id, email, name, created_at FROM member
WHERE created_at = (SELECT MIN(created_at) FROM member);

-- 다중 행 서브쿼리: 결과가 여러 개 → IN 연산자 사용
SELECT id, email, name, created_at FROM member
WHERE id IN (SELECT DISTINCT member_id FROM post WHERE member_id IS NOT NULL);
```

### 4.2 스칼라 서브쿼리 (SELECT 절)

각 행마다 단일 값을 계산해서 컬럼처럼 보여준다. JOIN 없이도 다른 테이블의 요약 정보를 끌어올 수 있다.

```sql
-- 각 게시글과 댓글 수를 함께 조회 (서브쿼리 방식)
SELECT p.id, p.title,
    (SELECT COUNT(*) FROM reply r WHERE r.post_id = p.id) AS reply_count
FROM post p;

-- 각 회원과 작성한 게시글 수 (서브쿼리 방식)
SELECT m.id, m.name,
    (SELECT COUNT(*) FROM post p WHERE p.member_id = m.id) AS post_count
FROM member m;
```

> 같은 결과를 LEFT JOIN + GROUP BY로도 만들 수 있지만, 스칼라 서브쿼리는 GROUP BY 없이 행 단위로 바로 계산값을 붙일 수 있어 간결하다.

### 4.3 FROM 절 서브쿼리 (인라인 뷰)

FROM 절의 서브쿼리는 **반드시 별칭이 필수**다. 별칭이 없으면 메인 쿼리에서 내부 컬럼을 참조할 수 없어 에러가 난다.

```sql
-- 회원별 게시글 수의 평균을 구하고, 그 평균보다 많이 쓴 회원 조회
SELECT member_id, COUNT(*) AS post_count
FROM post
GROUP BY member_id
HAVING COUNT(*) > (
  SELECT AVG(cnt)
  FROM (
    SELECT COUNT(*) AS cnt
    FROM post
    WHERE member_id IS NOT NULL
    GROUP BY member_id
  ) AS sub        -- 별칭(sub) 필수!
);
```

### 4.4 EXISTS 서브쿼리

서브쿼리 결과가 **한 건이라도 있으면 TRUE**를 반환. `IN`과 비슷하지만 존재 여부만 판단해서 대량 데이터에서 성능이 더 유리할 수 있다.

```sql
-- 게시글을 한 건이라도 작성한 회원만 조회
SELECT m.id, m.name, m.email
FROM member m
WHERE EXISTS (SELECT 1 FROM post p WHERE p.member_id = m.id);

-- 댓글이 달린 게시글만 조회 (NOT EXISTS로 반대 조건도 가능)
SELECT p.id, p.title
FROM post p
WHERE EXISTS (SELECT 1 FROM reply r WHERE r.post_id = p.id);
```

## 배운점

1. ✅ JOIN은 정규화로 나뉜 테이블을 다시 하나의 결과로 묶어주는 기법이다.
2. ✅ INNER JOIN은 양쪽 모두 매칭되는 행만(교집합), LEFT/RIGHT는 한쪽 테이블 전체 + 매칭 안 되면 NULL.
3. ✅ MySQL은 FULL OUTER JOIN을 지원하지 않아 LEFT JOIN + RIGHT JOIN을 UNION으로 우회한다.
4. ✅ SELF JOIN은 같은 테이블을 두 번 참조하는 것이라 별칭 구분이 필수다 (추천인, 상하관계 등).
5. ✅ UNION은 중복 제거(느림), UNION ALL은 중복 포함(빠름) — 합칠 컬럼 개수/타입이 같아야 한다.
6. ✅ 서브쿼리 결과가 1개면 `=`, 여러 개면 `IN`을 사용해야 한다.
7. ✅ 스칼라 서브쿼리(SELECT 절 안)는 GROUP BY 없이 행마다 집계값을 붙일 수 있어 간결하다.
8. ✅ FROM 절 서브쿼리(인라인 뷰)는 별칭이 없으면 에러 — 메인 쿼리가 내부 컬럼을 참조할 길이 없어진다.
9. ✅ EXISTS는 존재 여부만 검사해서 IN보다 대량 데이터에서 유리할 수 있다.

## 오늘 핵심 정리

```
JOIN 종류
  INNER JOIN       → 양쪽 다 매칭되는 행만 (교집합)
  LEFT OUTER JOIN  → 왼쪽 테이블 전체 + 매칭되는 오른쪽 (안 맞으면 NULL)
  RIGHT OUTER JOIN → 오른쪽 테이블 전체 + 매칭되는 왼쪽 (안 맞으면 NULL)
  FULL OUTER JOIN  → MySQL 미지원 → LEFT JOIN UNION RIGHT JOIN 으로 우회
  SELF JOIN        → 같은 테이블을 별칭 2개로 구분해서 자기 자신과 조인

집합 연산자
  UNION     → 합치고 중복 제거 (느림)
  UNION ALL → 합치고 중복 유지 (빠름)

서브쿼리 4종류
  WHERE 절   → 조건값 동적 계산 (단일값: =, 다중값: IN)
  SELECT 절  → 스칼라 서브쿼리, 행마다 단일값 계산
  FROM 절    → 인라인 뷰, 별칭(AS) 필수
  EXISTS     → 존재 여부만 TRUE/FALSE 판단
```
