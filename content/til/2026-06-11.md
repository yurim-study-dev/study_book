## 오전: 데이터베이스 기초 - 1.1

### 왜 데이터베이스가 필요한가?

프로그램 실행 중 변수나 객체에 저장된 데이터는 **RAM(휘발성)** 에 존재해서 프로그램이 종료되면 사라진다.
사용자 정보, 게시글, 거래 내역 같은 데이터는 **비휘발성 저장매체(HDD, SSD)** 에 영구 보존해야 한다.

```
메모리(RAM)      →   파일 시스템   →   DBMS
임시 저장(휘발)      영구 저장          체계적 관리
```

---

### 1.1.2 파일 시스템 방식의 한계

| 문제 | 설명 |
|------|------|
| **데이터 중복성** | 동일 데이터가 여러 파일에 중복 저장됨 |
| **데이터 불일치** | 한 파일 수정 시 다른 파일에 반영 안 됨 |
| **동시 접근 문제** | 여러 사용자 동시 수정 시 충돌 제어 어려움 |
| **데이터 종속성** | 파일 구조 변경 시 애플리케이션 코드도 함께 변경 필요 |

---

### 1.1.3 데이터베이스 4대 필수 요건

| 요건 | 설명 |
|------|------|
| **통합 데이터** | 중복 없이 한 곳에 통합 저장 |
| **저장 데이터** | HDD, SSD 등 디지털 저장 매체에 보관 |
| **운영 데이터** | 조직의 핵심 업무 처리에 반드시 필요한 데이터 |
| **공용 데이터** | 여러 사용자·시스템이 함께 공유 사용 가능 |

### 1.1.3 데이터베이스 4대 특징

| 특징 | 설명 |
|------|------|
| **실시간 접근성** | 조회 시 즉각적인 응답 |
| **계속적인 변화** | 추가·수정·삭제로 항상 최신 상태 유지 |
| **동시 공용** | 여러 사용자가 동시 접근해도 충돌 없이 안전 |
| **내용에 의한 참조** | 물리적 위치가 아닌 데이터 값으로 검색 가능 |

### 1.1.4 DBMS란?

**DBMS(Database Management System)**: 데이터베이스를 관리하고 사용자 요청에 따라 데이터를 조회/삽입/수정/삭제하는 소프트웨어

```
[응용 프로그램] ←→ [DBMS] ←→ [데이터베이스(물리적 저장소)]
```

- 응용 프로그램이 물리 파일에 직접 접근하면 데이터 손상 위험
- DBMS가 중간에서 모든 접근을 대행하고 안전하게 통제
- **핵심 기능**: 데이터 무결성 유지, 보안 제어, 동시성 제어, 장애 복구

---

### 1.1.5 주요 데이터베이스 유형 비교

#### 1) 관계형 데이터베이스 (RDBMS)

```
특징: 2차원 테이블 구조, 엄격한 스키마 정의 필수
장점: 데이터 무결성·일관성 최우선 보장, JOIN으로 복잡한 데이터 처리
단점: 구조 변경 어려움, 수평 확장 비용 큼
적합: 금융, 회원 관리, 주문·결제 (데이터 정합성이 중요한 서비스)
대표: MySQL, Oracle, PostgreSQL, MS SQL Server
```

#### 2) 비관계형 데이터베이스 (NoSQL)

```
특징: 스키마 없음, 자유로운 형태로 데이터 저장
장점: 구조 변경 유연, 대규모 트래픽·분산 처리에 강함
단점: JOIN 제한적, 일관성·정합성 보장 약함
적합: 실시간 채팅, 로그 수집, SNS 피드 (비정형 대용량 처리)
대표: MongoDB, Redis, Cassandra, Neo4j
```

#### 3) NewSQL & 특수 목적 DB

| 종류 | 설명 | 대표 제품 |
|------|------|---------|
| **NewSQL** | RDBMS 장점(ACID) + NoSQL 장점(수평 확장) 결합 | Google Spanner, CockroachDB |
| **벡터 DB** | 생성형 AI, 유사도 검색 특화 | Pinecone, Milvus |
| **시계열 DB** | 서버 모니터링, 센서 데이터 등 시간 흐름 데이터 | InfluxDB, Prometheus |

## 오전: RDBMS 구조 - 1.2

### 1.2.1 데이터베이스 구조 계층

```
데이터베이스 (board_db)
  └── 테이블 (member, post, reply ...)
        └── 컬럼 (id, email, name, created_at ...)
              └── 행/레코드 (실제 데이터 한 줄)
```

| 개념 | 설명 | 예시 |
|------|------|------|
| **데이터베이스** | 연관 테이블들의 논리적 보관 공간 | `board_db` |
| **테이블** | 행+열로 구성된 2차원 표 | `member`, `post`, `reply` |
| **컬럼(열)** | 데이터의 종류와 타입 정의 | `email VARCHAR`, `age INT` |
| **레코드(행)** | 실제 저장되는 데이터 한 줄 | 회원 한 명의 정보 |
| **스키마** | 테이블 구조·타입·제약조건의 명세 | `email은 문자열, 필수값` |

**레코드 예시 (member 테이블):**

| id | email | name | phone | address | created_at |
|----|-------|------|-------|---------|------------|
| 1 | haru@gmail.com | 하루 | null | 경기도 김포시 | 2026-06-11 |

### 1.2.2 기본 키(PK)와 외래 키(FK)

#### 기본 키 (Primary Key, PK)
- 각 행을 **유일하게 식별**하는 컬럼
- `NULL` 불가, 중복 불가

#### 외래 키 (Foreign Key, FK)
- **다른 테이블의 PK를 참조**하여 테이블 간 관계를 형성
- **참조 무결성**: FK 값은 참조 대상 테이블의 PK 값 중 하나이거나 null이어야 함

```
member 테이블          post 테이블
┌────────────┐         ┌─────────────────────┐
│ id (PK)    │◄────────│ id (PK)             │
│ email      │    FK   │ member_id (FK) ─────┘
│ name       │         │ title               │
└────────────┘         │ content             │
                       └─────────────────────┘
```

---

### 1.2.3 테이블 간의 관계

#### 1:N (일대다) - 가장 일반적인 관계

```
회원(1) ──────< 게시글(N)
  한 명의 회원이 여러 개의 게시글 작성 가능
  → N 쪽 테이블(post)에 FK 배치
```

#### N:M (다대다) 관계

```
학생(N) >────< 과목(M)
  한 학생이 여러 과목 수강, 한 과목을 여러 학생이 수강

→ RDBMS는 다대다를 직접 표현 불가
→ 매핑 테이블(연결 테이블)로 1:N + M:1 로 분해

학생(1) ──< 수강(N:M 매핑) >── 과목(1)
          student_id (FK)
          subject_id  (FK)
```

## 오후: DDL (Data Definition Language) - 2.1

DDL은 데이터베이스와 테이블의 **구조를 정의**하는 SQL이다. 실행하면 바로 반영되고 롤백이 안 되므로 주의가 필요하다.

### SQL 분류

| 분류 | 이름 | 역할 | 대표 명령 |
|------|------|------|---------|
| **DDL** | Data Definition Language | 구조 정의 | CREATE, ALTER, DROP, TRUNCATE |
| **DML** | Data Manipulation Language | 데이터 조작 | INSERT, SELECT, UPDATE, DELETE |
| **DCL** | Data Control Language | 권한 제어 | GRANT, REVOKE |

### CREATE DATABASE / USE / DROP DATABASE

```sql
CREATE DATABASE board_db;   -- 데이터베이스 생성
USE board_db;               -- 사용할 데이터베이스 선택
DROP DATABASE board_db;     -- 데이터베이스 삭제 (내부 테이블 모두 삭제됨 — 주의!)
```

### CREATE TABLE & 주요 데이터 타입

```sql
CREATE TABLE member (
    id         INT            AUTO_INCREMENT PRIMARY KEY,
    email      VARCHAR(100)   NOT NULL UNIQUE,
    password   VARCHAR(255)   NOT NULL,
    name       VARCHAR(100)   NOT NULL,
    created_at DATETIME       DEFAULT CURRENT_TIMESTAMP
);
```

**자주 쓰는 데이터 타입**

| 타입 | 설명 |
|------|------|
| `INT` | 정수 |
| `VARCHAR(n)` | 최대 n자 가변 문자열 |
| `CHAR(n)` | 고정 n자 문자열 |
| `TEXT` | 긴 텍스트 |
| `DATETIME` | 날짜 + 시간 (`2026-06-11 09:00:00`) |
| `BOOLEAN` | true / false (내부적으로 TINYINT(1)) |

### 주요 제약 조건 (Constraints)

| 제약 조건 | 설명 |
|-----------|------|
| `PRIMARY KEY` | 행 고유 식별자. NULL 불가, 중복 불가 |
| `AUTO_INCREMENT` | INSERT 시 값을 1씩 자동 증가 (INT PK에 주로 사용) |
| `NOT NULL` | 빈 값 삽입 불가 |
| `UNIQUE` | 중복 값 삽입 불가 (NULL은 허용) |
| `DEFAULT 값` | 값 미입력 시 지정된 기본값으로 자동 저장 |
| `FOREIGN KEY ... REFERENCES` | 다른 테이블의 PK를 참조 (참조 무결성 보장) |

### FK를 사용한 테이블 연결

```sql
CREATE TABLE post (
    id         INT          AUTO_INCREMENT PRIMARY KEY,
    userid     INT,
    title      VARCHAR(100) NOT NULL,
    content    VARCHAR(100) NOT NULL,
    created_at DATETIME     DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (userid) REFERENCES member(id)  -- member 테이블의 id를 참조
);
```

FK가 있으면 `member`에 존재하지 않는 `userid`는 INSERT 할 수 없어 데이터 무결성이 보장된다.

### DROP TABLE vs TRUNCATE

| 구분 | 설명 |
|------|------|
| `DROP TABLE 테이블명` | 테이블 구조 + 데이터 모두 삭제 |
| `TRUNCATE TABLE 테이블명` | 데이터만 전체 삭제, 구조(스키마)는 유지 |

```sql
DROP TABLE member;        -- 테이블 자체 삭제
TRUNCATE TABLE post;      -- post 테이블의 데이터만 초기화
```

## 오후: DML (Data Manipulation Language) - 2.2

DML은 테이블 안의 **데이터를 조작**하는 SQL이다.

### INSERT - 데이터 삽입

```sql
-- 단일 행 삽입
INSERT INTO member (email, password, name)
VALUES ('haru@gmail.com', '123', '하루');

-- 여러 행 한 번에 삽입
INSERT INTO member (email, password, name)
VALUES ('namu@gmail.com', '456', '나무'),
       ('yong@gmail.com',  '111', '용쌤'),
       ('harong@gmail.com','223', '하롱이');
```

> `AUTO_INCREMENT` 컬럼(id)과 `DEFAULT` 컬럼(created_at)은 생략 가능하다.

### SELECT - 데이터 조회

```sql
SELECT id, email, name, created_at FROM member;  -- 특정 컬럼
SELECT * FROM member;                             -- 전체 컬럼
SELECT title, content FROM post WHERE id = 5;    -- 조건 조회
```

### UPDATE - 데이터 수정

```sql
UPDATE member SET password = '456' WHERE id = 3;
```

> `WHERE` 절 없이 UPDATE 하면 **전체 행이 수정**되니 반드시 조건을 붙여야 한다.

### DELETE - 데이터 삭제

```sql
DELETE FROM member WHERE id = 1;
DELETE FROM post;   -- WHERE 없으면 전체 삭제 (TRUNCATE와 동일한 효과지만 롤백 가능)
```

## 오후: 실습 과제 - member / post / comments 테이블 CRUD

오늘 과제는 board_db 안에 3개 테이블을 만들고 각각 CRUD 를 수행하는 것이었다.

**테이블 관계**

```
member(1) ──< post(N) ──< comments(N)
  PK: id       FK: userid    FK: postid
                             FK: userid
```

**comments 테이블 생성 (FK 2개)**

```sql
CREATE TABLE comments (
    id         INT          AUTO_INCREMENT PRIMARY KEY,
    postid     INT,
    userid     INT,
    content    VARCHAR(100) NOT NULL,
    created_at DATETIME     DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (postid)  REFERENCES post(id),
    FOREIGN KEY (userid)  REFERENCES member(id)
);
```

댓글은 어느 게시글에 달린 것인지(`postid`)와 누가 썼는지(`userid`) 둘 다 참조한다. FK가 여러 개인 경우에도 각각 `FOREIGN KEY` 줄을 추가하면 된다.

## 배운점

1. ✅ 데이터베이스는 파일 시스템의 중복·불일치·종속성 문제를 해결하기 위해 등장했다.
2. ✅ DBMS는 응용 프로그램과 물리 저장소 사이의 중재자로, 직접 접근을 차단하고 안전하게 통제한다.
3. ✅ RDBMS는 무결성·일관성이 중요한 서비스에, NoSQL은 대용량 비정형 데이터 처리에 적합하다.
4. ✅ PK는 행을 유일하게 식별하고, FK는 다른 테이블의 PK를 참조해 테이블 간 관계를 형성한다.
5. ✅ DDL은 테이블 구조 정의, DML은 데이터 조작 — 역할이 완전히 다르다.
6. ✅ `AUTO_INCREMENT`는 PK 값을 자동으로 채워줘서 직접 지정하지 않아도 된다.
7. ✅ UPDATE/DELETE 에서 `WHERE` 없이 실행하면 전체 행이 바뀌니 항상 조건 확인 필수.
8. ✅ FK가 있으면 참조 대상 테이블에 없는 값은 삽입 자체가 거부된다 → 참조 무결성 자동 보장.

## 오늘 핵심 정리

```
SQL 3대 분류
  DDL  → 구조 정의 (CREATE, ALTER, DROP, TRUNCATE)
  DML  → 데이터 조작 (INSERT, SELECT, UPDATE, DELETE)
  DCL  → 권한 제어 (GRANT, REVOKE)

CREATE TABLE 핵심 옵션
  AUTO_INCREMENT  : PK 자동 증가
  NOT NULL        : 빈 값 금지
  UNIQUE          : 중복 금지
  DEFAULT 값      : 기본값 자동 설정
  FOREIGN KEY ... REFERENCES : 다른 테이블 PK 참조

DML 주의사항
  UPDATE / DELETE 에는 반드시 WHERE 조건 명시

DROP vs TRUNCATE
  DROP      → 테이블 자체 삭제
  TRUNCATE  → 데이터만 초기화, 구조 유지
```
