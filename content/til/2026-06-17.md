### 실습 문제 구성

**테이블 구조** — member / post / reply 3개 테이블, 외래키 관계

```
member ←── post ←── reply
         (ON DELETE SET NULL)   (ON DELETE CASCADE)
```

### 문제 1: 게시글 목록 조회

- 작성자 이름, 댓글 수 포함
- **24시간 이내** 게시글 → 제목 앞에 `(New)` 추가
- **댓글 3개 이상** 게시글 → 제목 앞에 `(Best)` 추가
- 최신 등록순 정렬

```sql
SELECT 
    p.id,
    CONCAT(
        IF(p.created_at >= NOW() - INTERVAL 1 DAY, '(New) ', ''),
        IF(COUNT(r.id) >= 3, '(Best) ', ''),
        p.title
    ) AS title,
    m.name AS writer_name,
    COUNT(r.id) AS reply_count,
    p.created_at
FROM post p
LEFT JOIN member m ON p.member_id = m.id
LEFT JOIN reply r ON p.id = r.post_id
GROUP BY p.id, p.title, m.name, p.created_at
ORDER BY p.created_at DESC;
```

> `IF()` 안에 `COUNT()` 집계 함수를 중첩 사용할 수 있다. `GROUP BY` 이후에 집계 결과가 확정되므로 `CONCAT`과 함께 사용 가능.

### 문제 2: 게시글 상세 조회

게시글 제목/내용/작성자 + 댓글 작성자 이름/댓글 내용을 한 번의 쿼리로 조회.

핵심 포인트: **같은 `member` 테이블을 두 번 조인** — 게시글 작성자와 댓글 작성자를 별칭으로 구분.

```sql
SELECT 
    p.title AS post_title,
    p.content AS post_content,
    m.name AS post_writer_name,
    m.email AS post_writer_email,
    rm.name AS reply_writer_name,
    r.content AS reply_content
FROM post p
LEFT JOIN member m ON p.member_id = m.id       -- 게시글 작성자
LEFT JOIN reply r ON p.id = r.post_id           -- 댓글
LEFT JOIN member rm ON r.member_id = rm.id;     -- 댓글 작성자 (member 두 번째 조인)
```

## 오후: 트랜잭션(Transaction) - 6

### 트랜잭션이란?

데이터베이스의 상태를 변경하는 여러 작업을 **하나의 논리적 단위**로 묶은 실행 흐름.

예시: A 계좌 → B 계좌 송금
- A 잔액 차감 (UPDATE)
- B 잔액 증가 (UPDATE)

두 작업이 **모두 성공하거나 모두 실패**해야 데이터 정합성이 유지된다. 둘 중 하나만 성공하면 큰일남.

---

### ACID 성질

트랜잭션이 안전하게 수행되기 위한 4가지 핵심 규칙.

| 성질 | 영문 | 설명 |
|------|------|------|
| **원자성** | Atomicity | 모두 성공 or 모두 실패 — 부분 성공 없음 |
| **일관성** | Consistency | 트랜잭션 후에도 제약 조건·규칙을 만족하는 유효한 상태 유지 |
| **격리성** | Isolation | 동시에 실행되는 트랜잭션들이 서로 간섭하지 않음 |
| **지속성** | Durability | 완료된 트랜잭션 결과는 장애가 생겨도 영구 보존 |

> **격리성 구현 방식**: 대부분의 RDBMS는 락킹 대신 **MVCC(Multi-Version Concurrency Control, 다중 버전 동시성 제어)** 기법을 사용 → 읽기와 쓰기를 동시에 처리하면서도 격리성 확보

---

### COMMIT & ROLLBACK

| 명령 | 설명 |
|------|------|
| `COMMIT` | 트랜잭션 내 모든 변경을 DB에 영구 반영. 다른 세션에서도 즉시 조회 가능 |
| `ROLLBACK` | 트랜잭션 시작 전 상태로 모든 변경 취소. 원자성 보장의 안전장치 |

---

### 트랜잭션 제어 SQL

```sql
-- 1. 트랜잭션 시작
START TRANSACTION;

-- 2. 비즈니스 쿼리 실행
DELETE FROM post WHERE member_id = 2;   -- 탈퇴 회원의 게시글 삭제
DELETE FROM member WHERE id = 2;        -- 회원 삭제

-- 3-a. 정상 완료 시 영구 반영
COMMIT;

-- 3-b. 오류 발생 시 원래 상태로 복구
ROLLBACK;
```

> `post.member_id`의 FK가 `ON DELETE SET NULL`이라서, 회원 삭제 시 게시글도 함께 삭제하려면 **회원 삭제 전에 게시글을 먼저 삭제**해야 한다.

---

### DBeaver 트랜잭션 모드 주의

| 모드 | 동작 |
|------|------|
| **Auto-Commit** (기본값) | DML(INSERT/UPDATE/DELETE) 실행 즉시 자동 반영 → ROLLBACK 불가 |
| **Manual Commit** | START TRANSACTION과 동일, 명시적 COMMIT/ROLLBACK 필요 |

수동 제어 방법:
- DBeaver: `Database > Transaction mode > Manual Commit`
- SQL: `START TRANSACTION` 명시적으로 실행

---

### 격리성 실습 (transaction-A / transaction-B)

두 개의 세션을 동시에 열어 격리성 동작을 확인.

- **세션 A** (`transaction-A.sql`): `START TRANSACTION` → DELETE 실행 → COMMIT/ROLLBACK
- **세션 B** (`transaction-B.sql`): `SELECT * FROM post` / `SELECT * FROM member`

세션 A가 COMMIT 전에는 세션 B에서 조회해도 변경 사항이 보이지 않음 → **격리성 확인**

## 배운점

1. ✅ 같은 테이블을 별칭으로 여러 번 JOIN할 수 있다 — 게시글 작성자, 댓글 작성자 동시 조회할 때 유용.
2. ✅ `CONCAT(IF(...), IF(...), 컬럼)` 패턴으로 조건부 접두사를 동적으로 붙일 수 있다.
3. ✅ 트랜잭션은 "모두 성공 or 모두 실패"를 보장하는 단위 — 데이터 정합성의 핵심.
4. ✅ ACID 중 격리성은 MVCC로 구현 — 락 없이 읽기/쓰기 동시 처리 가능.
5. ✅ DBeaver 기본값이 Auto-Commit이라서 ROLLBACK이 안 됨 — 실습 때 Manual Commit으로 바꿔야 한다.
6. ✅ FK가 ON DELETE SET NULL일 때 부모 삭제 전에 자식 데이터를 먼저 처리해야 의도한 결과가 나온다.

## 오늘 핵심 정리

```
게시판 종합 실습 핵심
  IF() 중첩 + CONCAT       → 조건부 태그 동적 생성 ((New), (Best))
  member 테이블 이중 JOIN  → 게시글 작성자 / 댓글 작성자 동시 조회
  LIKE '%검색어%'          → 제목 OR 내용 다중 조건 검색
  LIMIT n OFFSET m         → 페이징 (m = (페이지-1) × 개수)

트랜잭션 ACID
  Atomicity   → 모두 성공 or 모두 실패
  Consistency → 트랜잭션 후에도 제약 조건 유지
  Isolation   → 다른 트랜잭션 간섭 차단 (MVCC)
  Durability  → 완료된 결과는 영구 보존

TCL 명령어
  START TRANSACTION → 트랜잭션 시작
  COMMIT            → 영구 반영 (되돌릴 수 없음)
  ROLLBACK          → 트랜잭션 시작 전으로 복구
```
