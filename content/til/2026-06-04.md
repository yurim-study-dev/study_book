## 1. 정적(Static) 멤버 - 11.10

### Static이란?

**static**: 클래스 멤버로, 모든 인스턴스가 공유하는 메모리에 저장됨
- 객체 생성 없이 클래스명으로 직접 접근 가능
- 메모리 영역(Method Area)에 저장
- 프로그램 실행 시 클래스 로딩 때 메모리에 올라감

---

### Static 필드 (클래스 변수)

```java
public class MathUtil {
    // static 필드: 모든 인스턴스가 공유
    final static double PI = 3.14159;
    
    // 인스턴스 필드: 객체마다 독립적
    int r = 5;
}
```

**사용:**
```java
double area = MathUtil.PI * 5 * 5;  // 인스턴스 생성 없이 직접 접근
System.out.println("반지름 5인 원의 넓이는 " + area + " 입니다.");
```

---

### Static 메서드

```java
public class MathUtil {
    // static 메서드
    static int add(int n1, int n2) {
        return n1 + n2;
    }
}

// 호출
System.out.println("3 + 5 = " + MathUtil.add(3, 5));  // 객체 생성 필수 없음
```

**특징:**
- 인스턴스 생성 불필요
- 클래스명으로 직접 호출
- 대표적으로 `Math.random()`, `Arrays.toString()` 등

---

### Static 메서드의 제한사항

#### ❌ 인스턴스 필드/메서드 접근 불가

```java
public class MathUtil {
    int r = 5;              // 인스턴스 필드
    static double PI = 3.14159;  // static 필드

    static void area() {
        System.out.println(PI);     // OK: static 필드 접근 가능
        // System.out.println(PI * r * r);  // 컴파일 에러!
    }
}
```

**이유:**
- static 메서드는 객체 생성 전에 메모리에 올라감
- 인스턴스 필드는 객체 생성 시에만 존재
- 존재하지 않을 수 있는 필드에 접근 불가

---

#### main 메서드에서의 호출

```java
public class MainMethodTest {
    public void instanceMethod() {
        System.out.println("인스턴스 메서드 호출.");
    }

    public static void staticMethod() {
        System.out.println("static 메서드 호출.");
    }

    public static void main(String[] args) {
        // ✅ static 메서드는 직접 호출 가능
        staticMethod();  // OK
        
        // ❌ 인스턴스 메서드는 직접 호출 불가 (컴파일 에러)
        // instanceMethod();  // 에러!
        
        // ✅ 객체 생성 후에는 호출 가능
        MainMethodTest m = new MainMethodTest();
        m.instanceMethod();  // OK
    }
}
```

---

### Java 21 이후의 main 메서드 변화

**이전 방식 (Java 21 이전):**
```java
public static void main(String[] args) {
    // static이어야만 함
}
```

**새로운 방식 (Java 21 이후):**
```java
void main() {
    // static 없이도 가능
    // 메인 메서드만 특별히 처리
}
```

## 2. 캡슐화(Encapsulation) - 11.9 심화

### 캡슐화란?

**캡슐화**: 객체의 내부 상태를 보호하고, 외부 접근을 제한하는 기법

**목표:**
- 유효하지 않은 데이터 변경 방지
- 데이터 무결성 보장
- 메서드를 통한 제어된 접근

---

### Step 1: Private 메서드로 보호

```java
public class Score {
    int kor, eng, math;
    
    // private 메서드: 클래스 내부에서만 사용 가능
    private int sum() {
        return kor + eng + math;
    }
    
    // public 메서드: 외부에서 호출 가능
    double avg() {
        double result = Math.round(100 * sum() / 3.0) / 100.0;
        return result;
    }
}
```

**특징:**
- `sum()`은 private이므로 외부에서 직접 호출 불가
- 하지만 avg() 내에서는 호출 가능 (내부 호출)
- 외부는 `avg()`를 통해서만 평균 계산 가능

---

### Step 2: Private 필드 + Getter/Setter

```java
public class Score {
    // private 필드: 외부 직접 접근 차단
    private int kor;
    private int eng;
    private int math;
    
    // Getter: 값 읽기
    public int getKor() {
        return kor;
    }
    
    // Setter: 값 설정 (유효성 검사 포함)
    public void setMath(int math) {
        if (math > 100 || math < 0) {
            System.out.println("math 값은 0 ~ 100 사이로 입력해야 됩니다.");
        } else {
            this.math = math;
        }
    }
    
    // 생성자에서도 유효성 검사
    public Score(int kor, int eng, int math) {
        if (kor > 100 || kor < 0) {
            System.out.println("kor 값은 0 ~ 100 사이로 입력해야 됩니다.");
        } else {
            this.kor = kor;
        }
        // ... eng, math도 동일
    }
}
```

**사용:**
```java
Score haru = new Score(100, 90, 85);
haru.setMath(95);  // Setter로만 변경 가능
int math = haru.getMath();  // Getter로 값 읽기
```

**메모리 보호:**
```java
// ❌ 직접 접근 불가능 (컴파일 에러)
haru.kor = 1000;

// ❌ private 메서드 호출 불가능
haru.sum();  // 컴파일 에러
```

---

### Step 3: 실전 패턴 - 은행 계좌 관리

#### ✅ 좋은 예: 캡슐화 적용

```java
public class Account {
    // 잔고는 private으로 보호
    private int balance;
    
    // Getter만 제공 (읽기만 가능)
    public int getBalance() {
        return balance;
    }
    
    // 입금: 유효성 검사
    public void deposit(int amount) {
        if (amount > 0) {
            this.balance += amount;
            System.out.println(amount + "원이 입금되었습니다. 잔고: " + balance);
        } else {
            System.out.println("입금액이 잘못되었습니다.");
        }
    }
    
    // 출금: 잔고 확인
    public void withdraw(int amount) {
        if (balance >= amount) {
            this.balance -= amount;
            System.out.println(amount + "원이 출금되었습니다. 잔고: " + balance);
        } else {
            System.out.println("잔고가 부족합니다. 잔고: " + balance);
        }
    }
}
```

**사용:**
```java
Account myAccount = new Account();
myAccount.deposit(8000);    // 8000원이 입금되었습니다. 잔고: 8000
myAccount.withdraw(5000);   // 5000원이 출금되었습니다. 잔고: 3000
myAccount.withdraw(10000);  // 잔고가 부족합니다. 잔고: 3000

// ❌ 잘못된 접근 불가능
// myAccount.balance = -10000;  // 불가능 (private)
```

---

#### ❌ 나쁜 예: 캡슐화 없음

```java
public class AccountBad {
    // public 필드: 아무나 직접 변경 가능!
    public int balance;
    
    public void deposit(int amount) { ... }
    public void withdraw(int amount) { ... }
}
```

**문제점:**
```java
AccountBad bad = new AccountBad();
bad.balance = -10000;  // 잔고가 음수로! 데이터 무결성 깨짐
```

## 3. Static과 Instance 멤버 비교

| 구분 | Static 멤버 | Instance 멤버 |
|------|-----------|-------------|
| 선언 위치 | `static int x;` | `int x;` |
| 메모리 영역 | 메서드 영역 | 힙 영역 |
| 생성 시점 | 클래스 로드 시 | 객체 생성 시 |
| 접근 방법 | `클래스명.멤버` | `객체명.멤버` |
| 공유 여부 | 모든 인스턴스 공유 | 객체마다 독립적 |
| 생명 주기 | 프로그램 종료까지 | 객체 삭제까지 |

**메모리 그림:**
```
메서드 영역:
┌──────────────────┐
│ PI = 3.14159     │ ← static 필드 (한 번만 생성)
│ add() 메서드     │ ← static 메서드
└──────────────────┘

힙:
┌──────────────────┐
│ r = 5            │ ← instance 필드 (객체1)
└──────────────────┘
┌──────────────────┐
│ r = 10           │ ← instance 필드 (객체2)
└──────────────────┘
```

## 4. 캡슐화 체크리스트

### Private 필드 원칙

```
필드는 private으로 보호
  ↓
필요하면 Getter/Setter 제공
  ↓
Setter에서 유효성 검사
```

### 예: 나이 설정

```java
private int age;

public int getAge() {
    return age;
}

public void setAge(int age) {
    if (age < 0 || age > 150) {
        System.out.println("유효한 나이가 아닙니다.");
    } else {
        this.age = age;
    }
}
```

## 배운점

1. ✅ static 멤버는 객체 생성 없이 클래스명으로 접근 가능하다.
2. ✅ static 메서드는 인스턴스 필드와 메서드에 직접 접근할 수 없다.
3. ✅ main 메서드는 static이므로 인스턴스 메서드는 객체 생성 후 호출해야 한다.
4. ✅ Java 21부터 main 메서드는 static 없이도 선언 가능하다.
5. ✅ 캡슐화는 필드를 private으로 보호하고 메서드로 제어하는 기법이다.
6. ✅ Getter/Setter를 통해 유효한 데이터만 저장할 수 있다.
7. ✅ 데이터 무결성을 위해서는 public 필드를 피하고 private + 메서드를 사용해야 한다.
8. ✅ Private 메서드로 외부에 공개할 필요 없는 기능을 은닉할 수 있다.
9. ✅ 입금/출금 같은 비즈니스 로직은 메서드로 제어해야 데이터 보호가 가능하다.
10. ✅ static은 메모리 절약, 캡슐화는 데이터 보호라는 서로 다른 목표를 가진다.
